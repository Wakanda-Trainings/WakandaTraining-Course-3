Wakanda Training Day 3 - Security


How to keep bad things from happening

Wakanda Security - Know Your Options

  • Scope (Public vs. Public on Server)
  • Users and authentication (who is the user?)
  • Groups and permissions (what can/can't the user do?)
  • Restricting Queries (which entities can the user access?)
  • Class Methods
  • Calculated attributes
  • Dataclass events
  • Session management
  • Inheritance
  • Transactions
  • HTTPS

Wakanda Security Suggestions

  • Design your model so that data and functions that should
    never be accessible to the browser are only available to
    code running on Wakanda Server
  • Place as much business logic as possible in the
    code that is only accessible on the server
  • Use authentication and permissions to assure that
    users can only interact in a proper way with dataclasses
    and methods
  • Wrap grouped modifications into class methods with
    parameters so that everything can be saved in one
    transaction

Wakanda Security Suggestions

  • Use restricting queries to constrain access to entities
    when needed
  • Don’t assume that attributes you provide for ‘display only’
    won’t be altered
  • Construct your client-side code so that it supports and mirrors
    the server-side restrictions but don’t count on client-side
    code to block a user action
  • Reference your pages through HTTPS so that communication with the
    server is encrypted

Scope

  • Allows you to restrict portions of your data model from client-side access
  • Cannot be changed at run time so very secure
  • Permissions oblivious - doesn't care about the current user's permissions
  • Can be applied to attributes, methods and entire dataclasses
